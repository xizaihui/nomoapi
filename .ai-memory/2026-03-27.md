## Channel Affinity 分析

### 系统已有架构 (不需要重写!)
- `service/channel_affinity.go` (950行) — 完整的渠道亲和性系统
- `setting/operation_setting/channel_affinity_setting.go` — 规则配置
- `middleware/distributor.go` — 调度入口
- 已有规则: Codex CLI (prompt_cache_key) + Claude CLI (metadata.user_id)

### 关键架构要点
- Channel Affinity 是**渠道级**亲和性，不是 key 级
- HybridCache: 内存 LRU (10万条) + Redis 双层
- `affinityFingerprint()` 用 SHA1 hash 取前 8 位
- 成功后才记录 (SwitchOnSuccess)
- fallback 到 `GetRandomSatisfiedChannel()` (优先级+权重)

### 讨论结论
- **system prompt 前 500 字截断 hash 方案不可行**: Claude Code/Cursor 等工具的 system prompt 前几千字符都是相同模板，无法区分不同用户
- **正确的亲和性 key**: 按 token/用户维度 (`context_int:id`)，同一用户 system prompt 通常固定
- **号池方案**: 拆渠道 (每 key 一个渠道) 优于改 key 选择逻辑
- 待用户决定是否添加通用 token 亲和性规则

### 渠道选择完整链路
```
Distribute() → GetPreferredChannelByAffinity()
  → 规则匹配 (model_regex + path_regex + key_sources)
  → extractChannelAffinityValue (gjson/context_int/context_string)
  → HybridCache 查询 → 命中返回 channel ID
  → 未命中 → CacheGetRandomSatisfiedChannel()
    → GetRandomSatisfiedChannel() (优先级分层 + 加权随机)
  → 成功后 RecordChannelAffinity() 写缓存
```
