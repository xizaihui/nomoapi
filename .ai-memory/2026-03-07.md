# Nexus Cloud Control — 项目总结

## 1. 项目名称和目标

**项目名称：** Nexus Cloud Control（云手机控制系统）

**目标：** 通过安装在云手机上的 Android APK（NexusAgent）与部署在服务器上的 Go 后端（cloud_control），实现对多台云手机的集中管理和远程控制。核心场景是批量管理运行 Suno 音乐应用的云手机集群。

**架构：**
- 云手机端：NexusAgent APK → 每 5 秒心跳上报状态 + 接收并执行云端指令
- 服务器端：Go + Gin Web 服务 → 云控页面展示设备状态 + 下发控制指令
- 通信方式：APK 主动 POST 心跳，服务端在心跳响应中携带待执行命令（拉模式）

## 2. 已完成的功能

### APK 端（NexusAgent v2.0.0）

| 功能 | 说明 |
|------|------|
| 心跳上报 | 每 5 秒上报：Suno 状态、脚本 PID、证书状态、脚本配置参数 |
| Suno 状态检测 | 三重检测（ps -ef / pidof / dumpsys activity），解决误报问题 |
| 远程启动 Suno | 4 级策略自动尝试（am start → monkey → 无指定 Activity → 自动查找 launcher） |
| 远程停止 Suno | am force-stop + kill -9 + 验证 |
| 远程启动脚本 | 接收 threshold/interval 参数，生成并启动 autotap 脚本 |
| 远程停止脚本 | 查找并 kill 脚本进程 |
| 远程注入证书 | mount rw + cp + chmod |
| 设备自注册 | 启动时自动调用 /api/agent/register，服务端自动写入 accounts.json |
| su 模式自动探测 | 自动检测 root 环境：直接 root / su -c (Magisk) / su 0 sh -c (toybox) / 回退 sh |
| 开机自启 | BootReceiver 监听 BOOT_COMPLETED，恢复上次运行状态 |
| 保活机制 | 前台通知 + WakeLock + START_STICKY + 忽略电池优化 |
| 完整配置界面 | 服务器地址、账号别名、计费模式、设备别名、IP端口、ADB密码 |

### 服务端（main.go）

| 功能 | 说明 |
|------|------|
| /api/agent/heartbeat | 接收心跳，双向映射 IP↔别名，响应中携带命令 |
| /api/agent/register | 设备自注册接口，自动更新 accounts.json |
| agentIsAlive() | 统一的 Agent 在线检测（同时查 IP 和别名） |
| agentGetStatus() | 统一的状态获取（同时查 IP 和别名） |
| agentSendCommand() | 统一的命令下发（双写 IP 和别名队列） |
| resolveAliasToIP() | 别名→IP 反查，从 accounts.json 配置中解析 |
| 所有操作函数已统一 | internalStartScript/StopScript/StartSuno/StopSuno 均使用统一辅助函数 |

## 4. 未完成 / 需要优化的内容

- **Suno 启动的 tap 坐标是硬编码的**（536,1841 和 385,297），不同分辨率的设备可能需要调整，可以考虑改为服务端下发坐标或按屏幕比例计算
- **accounts.json 的 api_key 字段**：自注册时生成的是时间戳 hex，如果有实际用途需要改为真正的 API key
- **APK 没有 release 签名**：当前是 debug 签名，正式部署建议配置 release keystore
- **服务端没有 HTTPS**：心跳和注册接口走的是 HTTP 明文
- **日志持久化**：APK 的日志只在内存中保留 8000 字符，重启后丢失，可以考虑写文件
- **批量操作确认**：云控页面的批量操作（如全部停止 Suno）通过 Agent 下发时没有结果回调确认机制
- **心跳间隔不可配置**：固定 5 秒，可以考虑服务端动态调整
